# OKF 知识库方案核心

## 一句话定义

> 在钉钉中用纯文本文档建立一层"知识对象元数据 + 原始证据"的双层架构，外部 Agent 通过 dws CLI 检索元数据定位原文，最终回答带引用。

## 双层职责

```
OKF 元数据卡片                原始文档
  ↓                            ↓
描述知识对象                 完整业务背景
标记当前状态 (active/superseded/draft) 详细规则
标记适用范围                 历史方案
记录别名                     评审与决策记录
关联原文 doc_id              证据来源
描述版本继承
```

**回答原则**：OKF 卡片用于定位与概括，原始证据用于支撑结论。

## 钉钉知识库目录结构

```
<目标知识库>
├── 00_MANIFEST       全局/分类清单（manifest-all + 按类型拆分）
├── 01_INDEX          总索引、产品索引、原文索引
├── 02_PRODUCT        产品事实卡片（每个产品 1 张）
├── 03_CAPABILITY     产品能力 / 模块卡片
├── 04_RULE           当前有效业务规则
├── 05_RESOURCE       金融资源 / 第三方供应商等（如适用）
├── 06_DECISION       关键决策与路线选择
├── 07_SERVICE        底层服务、接口、技术能力
└── 08_SOURCE         原始 PRD / 设计 / 实施计划 / 评审记录
    ├── <产品域 1>/
    ├── <产品域 2>/
    └── <PRD 归档>/
        └── _原始docx/    （可选 docx 附件存档）
```

## OKF 卡片字段规范

详见 [okf-spec.md](./okf-spec.md)。

## 标准检索流程（外部 Agent 必须遵循）

```
1. 用户问题 → 抽取实体 / 类型 / 时间约束
2. 读 manifest（00_MANIFEST/manifest-all 或按类型拆分版）
3. 在 manifest 里 grep 关键词 + 别名 → Top-N 候选 OKF id
4. 读 Top-1 OKF 卡片 → 看 status / effective_from / sources
5. status = active 才作为当前结论
6. 沿 sources 里的 doc_id 用 dws doc read 拉原文
7. 检查原文是否冲突
8. 生成回答 + 引用（doc_id + updated_at）
```

## 核心规则

1. **不要直接搜原文**：钉钉原生搜索把原文 PRD 排在前面，OKF 卡命中率低。Agent 必须先读 manifest 走结构化路径。
2. **status 字段是过期事实的隔离机制**：`superseded` / `deprecated` 卡片不能作为当前结论使用，只作为历史背景或版本继承的引用。
3. **aliases 字段是召回率关键**：决定能不能命中"用户随口怎么问"。卡片作者要预先把正式名 / 俗称 / 历史名 / 英文 / 简写都列上。
4. **summary 写 2-3 行不要写 1 行**：高区分度关键词（产品名、版本号、核心动词）越多越好命中。
5. **多份原文同一对象 → 1 张卡片 + 多 sources**：不要为每份 PRD 都产单独卡片。

## 评测指标（PoC 必跑）

1. **元数据定位成功率** Top-1/3/5
2. **原文定位成功率**
3. **回答准确率**（事实/范围/引用是否齐全）
4. **过期事实误用率**（`superseded` / `deprecated` 是否被错误当成当前结论）
5. **权限正确率**（多角色用户测试越权）
6. **检索成本**（CLI 次数 / 读文档数 / token）

最关键指标：**过期事实误用率应接近 0**。
